1: Algorithm

- The .s_algo128p file

- Advanced sorting

- Content detection

The .s_algo128p file

The .s_algo128p file is a 128 bit SNU algorithm Python file. There are many of these files, each one acts as an algorithm for a certain type of content. When processing a video, normally there can be up to 10000 different algorithm files scanning at once.

For example, if someone were to upload an image or video of a dog, an algorithm file would have to notice this. It would then fully activate if true. It will then get info on what type of dog it, and more. You can go more into depth with this with the advanced sorting section.

The file uses Python syntax. A new algorithm file needs to be created for each media type, and needs to be constantly updated.

Advanced sorting

SNU uses a very advanced sorting mechanism when content is put on the site. The site has to gather as much info as possible for every pixel of content. An advanced and efficient sorting process is put into place.

When content is uploaded to the site, detection is put into place so that it can be placed in the right areas of the site. Illegal content, such as Child Pornography, murder, and more must be placed in the evidence locker, which can then be used for trial against the uploader.

2: Daily updates

- Schedule

What is a documentation update

A documentation update is an update to the website that is done to keep development active on a daily basis. Very basic documentation updates include only updating the update and developer logs, which adds very little to the site.

An update is no longer considered just a documentation update if a feature is updated that isn't purely for documentation. Some documentation updates are good, and they can be used for Sean to quickly make an update to keep the site active, but they shouldn't be the only update released, however there have been issues in the past regarding this, including a long period of documentation updates during the end of the 3.0.4 development phase, waiting to add features for 3.0.5

Update schedule

Sean has been working on SNU all by himself for a few years now. Starting in December 2017, it was written in Google sites, but SNU wasn't there for a while. The Google Sites version was eventually abandoned. In late 2018, a horribly made JavaScript version was made, but was not functional. It was simply taking notes.

However, the biggest change happened in January of 2019. While experimenting with HTML hyperlinks, Sean found out how to link between files. This led to HTML discovery and learning, and the first versions of SNU 3 were created. Over time, this came to be the biggest release of SNU to date. Many many improvements were made, and the concept was heavily tweaked.

With SNU 3.0, 3.0.1, 3.0.2, and 3.0.3, development was at random. Experiments were made daily, and features were tested until Sean moved on. Starting with version 3.0.4, updates were slowed down, with major releases coming less frequently, the current branch would be worked on. Sean tried to keep development going on a daily basis.

Starting with SNU 3.0.5, the site changed to a twice a year release model. The first version would be released on June 14th xxxx and the second version would be released on December 14th xxxx. The December 2019 update has not yet occurred, but the schedule is still solid.

This is done to prevent lots of memory being used for updates. A major site update is a copy of the site directory, and starting with 3.0.5, a major change was incorporated with each major update. Within each update, there are daily updates. An update is defined as when a file is created or modified. For example, if a snapshot was created, that snapshot alone counts as an update, while the page modifications count as 1 update separately.

Updates are created on a daily basis, with new changes daily. If a feature fails to be added, a documentation update is created instead to keep development active.

1D mode concept

The purpose of 1D mode is to be a text only part of the website. Images, video, audio, and other media is not allowed here. This mode is also made to be more bare bones and work with slower internet connections.

1D mode pages never go above 100 kilobytes in size, making them suitable for slower internet connections, such as DSL, or dialup. If you just want to see text and you have good internet, 1D mode is the best option.

1D mode limits

The limtis of 1D mode are

1D Mode browser compatibility

Since 1D mode is written in all forms of HTML, and KHTML, it is compatible with most web browsers. It was designed to work on any web browser, even the first web browser. 1D mode is very light and for this reason, older web browsers on slow connections can load pages normally. It is like being in the early 1990s with a dialup connection, but instead of getting forced off the internet when a phone call happens, you can just use a virtual machine, and configure it correctly.

It is not recommended to use 1D mode on any version of a web browser from before 2012 without being in a virtual machine. This means you shouldn't use Firefox 4 on your main machine to run 1D mode, you shouldn't use Internet Explorer 6 on your machine in 1D mode, and so on. For extra security, you can try being in a virtual machine inside a virtual machine.

1D mode speed

Due to 1D modes lightly written scripting, it is an extremely easy to load set of pages. A modern internet connection can load a 1D mode page in less than a millisecond. Even 3G would be fast for 1D mode. It is recommended to have a connection of at least 40 KB/s to run 1D mode fast and smoothly

11: Servers, subservers, superservers, and megaservers

Brief summary

SNU is not just a single server. There is a lot that goes into SNU, and the term "server" doesn't just refer to the main server.

The main server is a superserver, it hosts the main content by the main SNU team. Within the superserver, there are subservers. A subserver example would be the VoIP service. Users can create their own subservers in the superserver.

The Megaserver is the grand scheme of all SNU servers around the known universe together. When you download the SNU server software, you are hosting a secondary superserver within the megaserver.

Starting a server

For a user to start their own SNU server, they are going to need to download a stable version of the site, along with the server essentials tool kit. 13: Site security

Brief summary

The security of SNU is vital. We can't have servers getting compromised, systems being abused, or lives affected by compromising accounts. Several security techniques are going to be put into place to make sure the SNU ecosystem is strong. Someday in the future, possibly unhackable

If you are going to hack the site to test its security, we strongly support this, however please tell us first. Every time the server is attacked, another vulnerability will be patched, and if it is attacked enough, it could become undestructable.

- SQL Security methods

-

SQL security methods

SNU uses SQL for searches. However, there is a risk of an SQL injection attack if not properly programmed.

On August 25th 2019, Sean realized something. Most SQL injection attacks confuse the server into logging into the first account on the site. SNU accounts don't use SQL, but some sections of the site may allow it. The idea that he came up with is that if they were able to break into the first account, instead of patching it, the first account would be a locked account that lets you view the site, but nothing else. No other interaction.

SQL security flaws will be patched quickly once found.
